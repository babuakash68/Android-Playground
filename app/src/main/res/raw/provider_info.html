1. Content Provider Structure

• PillarsContentProvider extends ContentProvider

• Uses an in-memory list to store contacts

• Implements CRUD operations (Create, Read, Update, Delete)

2. Data Access Flow

Query Operation

→ When you click "View List", the app calls contentResolver.query()

→ PillarsContentProvider.query() is invoked

→ Returns a Cursor with all contacts

→ Activity processes the Cursor to display contacts

Insert Operation

→ When you click "Add Contact", the app calls contentResolver.insert()

→ PillarsContentProvider.insert() is invoked

→ New contact is added to the in-memory list

Delete Operation

→ When you click "Erase All", the app calls contentResolver.delete()

→ PillarsContentProvider.delete() is invoked

→ All contacts are removed from the in-memory list

Update Operation

→ When you edit a contact, the app calls contentResolver.update()

→ PillarsContentProvider.update() is invoked

→ Contact details are updated in the in-memory list

3. Device Contacts Access

• App requests READ_CONTACTS permission

• Uses ContentResolver to query device contacts

• Imports contacts into our Content Provider

4. Data Persistence

• All data is stored in memory only

• Data is cleared when the app is closed

• No data is shared with other applications
